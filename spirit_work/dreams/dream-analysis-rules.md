# Dream analysis rules

Point the assistant at this file when writing or revising a dream entry in `spirit_work/dreams/`.

These rules describe the format and voice already in use. Prefer recent files (`july14.md`, `jun22.md`, `may30.md`) over early emoji-heavy ones when styles conflict.

Also respect `guide.md` top rules (no regurgitation, full sentences, plain English). Dreams live here, not in `days/`.

---

## File location and naming

- Put dream accounts and analysis only in `spirit_work/dreams/`.
- One file per date (or night), named like `july14.md`, `may30.md`.
- Do not put dream analysis into `days/*.md`.

---

## Document structure

Use this order unless the user asks for something else:

1. **Title block**
   - `# 📅 MM-DD`
   - Bold subtitle naming the night’s main images/themes (not a generic “Dream Analysis” only)

2. **`## The dream`**
   - Second person (“you”)
   - Organize the user’s raw account into clear scenes
   - Use `###` scene headings when the night has multiple parts
   - Keep uncertainty (“you do not remember…”, “you think…”)
   - Put waking context in labeled notes when the user supplies it, e.g. `**Waking note on …:**`
   - Fix grammar/spelling in the dream write-up without changing meaning

3. **`---` then `## Analysis`**
   - Go straight into the analysis. Do not open with meta reminders about the rules, the dream section, or not restating the plot
   - **Analyze this dream first.** Mechanisms and waking moves come before any comparison to other files
   - Keep the analysis substantial but not padded. Say enough to be useful. Do not repeat the same claim in three forms. Do not strip the meaning down to empty slogans.
   - End with a concrete **What to do with it:** line when the read supports action
   - **Only after** the analysis, add a short related-notes section if other dreams or `days/` entries truly help — optional, secondary, brief

4. **Optional closing sections** (use as needed)
   - `## Follow-up reflections` or questions — prompts with `**You:**` / `**Assistant:**` blanks for later dialogue
   - `## Usable analysis (kept after cleanup)` — plain claims version after a draft got too plot-heavy

---

## Analysis voice (non-negotiable)

1. **Do not regurgitate the dream.**  
   The dream section already holds what happened. Analysis only adds mechanism, meaning, and waking moves. Do not restate scenes, counts, uniforms, nations, dialogue, or plot beats the user already wrote. If a sentence could be cut and the user would still recognize their own account, cut it. Wrong: “United States and German personnel are in the same space, and the uniforms look enough alike…” Right: “Rank is obvious. Affiliation is not. Helpful certainty runs ahead of a clean read.”

2. **Full sentences only.**  
   Write complete sentences that make sense on their own. No sentence fragments. No run-on sentences. Split long thoughts into separate sentences. Prefer clear subject–verb–object over stacked clauses.

3. **Plain words, real meanings.**  
   Do not be cute with language. Do not stretch, bend, or reinvent a word to sound clever. Use each word for what it actually means. If a phrase needs explaining to work, rewrite it in ordinary English.

4. **Mechanism over poetry.**  
   Prefer plain claims over symbolic theater. No opaque metaphor stacks, no fake jargon, no invented esoteric terms for a visual that already speaks (e.g. a hooked face = power that grabs).

5. **Second person.**  
   Write to “you.”

6. **Analyze this dream first.**  
   The main job is what this night is doing. Comparisons to other dreams or day files come after, only if useful, and stay short.

7. **Do not define a dream by what it is not.**  
   Do not spend analysis explaining that it is not a maze dream, not a war dream, or not some other type. Say what it is doing. If the user rejects a label, drop the label. Do not write a rebuttal section about the rejected label.

8. **Do not force a series.**  
   Only link another dream when the link is clear and helpful. Never stretch a night into a family it does not belong in.

9. **Sequence over dictionary symbols.**  
   Prefer positions and moves over isolated “X means Y” glossaries.

10. **Tie to waking life with evidence.**  
    After the dream analysis, you may link recent `days/` entries or named patterns the user already stated. Keep dream language primary.

11. **Spirit content can be practical.**  
    When the user supplies spirit/entity history, treat it as usable information (doors to keep closed, fields that hitch), not only metaphor — and do not invite them back into work they already stopped.

12. **Actionable close.**  
    Prefer one concrete waking move over a long interpretive flourish. “What to do with it” should be doable this week when possible.

13. **Repetition means the pattern is still true.**  
    Recurring dreams usually persist because the waking structure is still active — not because one hidden clue was missed.

14. **Leave honest gaps.**  
    Do not invent missing plot, names, or motives. Say what the dream showed and what it left open.

15. **Honor the user’s own read.**  
    If they already interpreted a piece, build on it or stress-test it; do not overwrite it with a cleverer theory.

16. **No coincidence framing** (per `guide.md`). Treat meaningful links as links.

---

## What analysis is for

- Say what **this** night is doing.
- Offer a **usable next move**, not a fortune or a prophecy.
- Keep it substantial enough to be useful, without padding or empty slogans.
- Add comparisons only after the analysis, and only when they help.

## Anti-patterns (do not do)

- Regurgitating the dream plot, including paraphrases of what the user already wrote
- Opening analysis by summarizing the plot again
- Opening analysis by restating process rules or meta reminders to the user
- Cute synonyms that change what the user said (e.g. “sideways” for “did not answer directly”)
- Collapsing distinct facts into one wrong abstraction (e.g. calling similar uniforms “roles”)
- Stripping the analysis so thin it has no usable meaning
- Padding the analysis by repeating the same point
- Defining the dream by what it is not
- Leading with comparisons to other dreams before analyzing this one
- Forcing the dream into a series it does not belong in
- Long analysis with many subsections when a short read would do
- Sentence fragments or run-on sentences
- Clever wordplay that changes what a word actually means
- Symbol-dictionary dumps with no waking link
- Emphasizing poetic atmosphere over what the dream is doing
- Putting dream write-ups in `days/`
- Logging AI meta-rules into the dream file (those belong here or in `guide.md`)
- Claiming certainty the dream did not earn
